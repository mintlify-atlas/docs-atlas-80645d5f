## Overview

The Click Creators Scraper Client uses a **multi-tenant architecture** to ensure complete data isolation between different scraping jobs. Each job operates in its own isolated namespace, preventing data leakage and enabling parallel campaigns.

## What is Multi-Tenancy?

Multi-tenancy is a software architecture where a single instance of the application serves multiple customers (tenants), with each tenant's data completely isolated from others.

**In Click Creators:**
- Each **scraping job** is a tenant
- The **base_id** (Airtable base ID) serves as the tenant identifier
- All data tables use `base_id` to filter and isolate data

<Note>
Think of `base_id` as a "workspace ID". Just like different teams in an organization have separate workspaces, each scraping job has its own isolated data space.
</Note>

## Why Multi-Tenant Architecture?

### Benefits

1. **Data Isolation:** Profiles from Campaign A never appear in Campaign B
2. **Parallel Campaigns:** Run multiple influencer campaigns simultaneously
3. **Client Separation:** Manage campaigns for different clients without overlap
4. **Easier Debugging:** Issues in one job don't affect others
5. **Flexible Scaling:** Add new jobs without database schema changes

### Real-World Scenario

```
Job 1: Nike Instagram Campaign
├── base_id: app1a2b3c4d5e6f7g8
├── 50,000 scraped profiles
├── 80 VAs in Airtable
└── Daily assignments

Job 2: Adidas TikTok Campaign  
├── base_id: app9z8y7x6w5v4u3t
├── 30,000 scraped profiles
├── 40 VAs in Airtable
└── Daily assignments
```

These two jobs run **completely independently** despite sharing the same database and application instance.

## The base_id System

### What is base_id?

From `lib/scraping-jobs.ts:35`:
```typescript
const { data } = await supabase
  .from('scraping_jobs')
  .insert({
    influencer_name: input.influencer_name,
    platform: input.platform,
    airtable_base_id: input.airtable_base_id,
    num_vas: input.num_vas,
    base_id: input.airtable_base_id  // Use airtable_base_id as tenant ID
  })
```

**Key Points:**
- `base_id` is **derived from** Airtable base ID
- Format: `app` + 16 alphanumeric characters (e.g., `app1a2b3c4d5e6f7g8`)
- Generated by Airtable when you create a base
- Immutable once set (never changes for a job)

### Source of Truth: BaseContext

The `base_id` is managed globally via React Context:

From `contexts/base-context.tsx:11-23`:
```typescript
interface BaseContextType {
  activeJob: ScrapingJob | null
  baseId: string | null  // Source of truth from active job
  airtableBaseId: string | null
  isLoading: boolean
  error: string | null
  refreshActiveJob: () => Promise<void>
  setActiveJobById: (jobId: string) => Promise<void>
  clearActiveJob: () => void
}
```

**How it works:**
1. User selects a job from sidebar or URL (`?job={job_id}`)
2. BaseContext fetches the job from `scraping_jobs` table
3. Extracts `base_id` from the job record
4. All components use this `base_id` for database queries

<Tip>
The `base_id` is **ALWAYS** fetched from the database, never hardcoded or cached locally. This prevents stale tenant IDs.
</Tip>

## Multi-Tenant Database Schema

Every data table includes a `base_id` column for tenant filtering:

### scraping_jobs
```sql
CREATE TABLE scraping_jobs (
  job_id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  influencer_name TEXT NOT NULL,
  platform TEXT NOT NULL,
  airtable_base_id TEXT,
  base_id TEXT NOT NULL,  -- Tenant identifier
  num_vas INTEGER,
  status TEXT DEFAULT 'active',
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### source_profiles
```sql
CREATE TABLE source_profiles (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  username TEXT NOT NULL,
  base_id TEXT NOT NULL,  -- Isolates profiles by job
  created_at TIMESTAMP DEFAULT NOW()
);
```

### global_usernames
```sql
CREATE TABLE global_usernames (
  username TEXT PRIMARY KEY,
  used BOOLEAN DEFAULT FALSE,
  base_id TEXT NOT NULL,  -- Isolates username pool by job
  full_name TEXT,
  job_id UUID,
  created_at TIMESTAMP DEFAULT NOW()
);
```

### daily_assignments
```sql
CREATE TABLE daily_assignments (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  campaign_id UUID REFERENCES campaigns(campaign_id),
  username TEXT,
  full_name TEXT,
  va_table_number INTEGER,
  position INTEGER,
  job_id UUID,
  base_id TEXT NOT NULL,  -- Isolates assignments by job
  status TEXT DEFAULT 'pending',
  assigned_at TIMESTAMP DEFAULT NOW()
);
```

### campaigns
```sql
CREATE TABLE campaigns (
  campaign_id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  campaign_date DATE NOT NULL,
  total_assigned INTEGER NOT NULL,
  base_id TEXT NOT NULL,  -- Isolates campaigns by job
  status TEXT CHECK (status IN ('pending', 'success', 'failed')),
  created_at TIMESTAMP DEFAULT NOW()
);
```

<Warning>
Every table that stores job-specific data **must** include a `base_id` column. Missing this column breaks tenant isolation.
</Warning>

## Context-Aware Supabase Client

### Standard Client (No Context)

```typescript
import { supabase } from '@/lib/supabase'

// Queries ALL rows (no tenant filtering)
const { data } = await supabase
  .from('global_usernames')
  .select('*')
```

### Context-Aware Client

From `lib/supabase.ts`:
```typescript
export function createSupabaseClientWithContext(baseId: string) {
  return supabase.rpc('set_base_context', { base_id: baseId })
}
```

Usage:
```typescript
// Create client with base_id context
const supabaseWithContext = createSupabaseClientWithContext(baseId)

// Query with automatic tenant filtering
const { data } = await supabaseWithContext
  .from('global_usernames')
  .select('*')
  .eq('base_id', baseId)  // Belt-and-suspenders approach
```

<Note>
The "belt-and-suspenders" approach uses **both** the context-aware client AND an explicit `.eq('base_id', baseId)` filter. This provides defense-in-depth against data leakage.
</Note>

## Job Selection and Context Sync

### URL-Based Job Selection

From `app/callum-dashboard/page.tsx:41-72`:
```typescript
const jobId = searchParams.get('job')

// Sync BaseContext with URL job parameter
useEffect(() => {
  if (jobId) {
    console.log('Syncing BaseContext with job from URL:', jobId)
    setActiveJobById(jobId)  // Fetches job and updates baseId
  }
}, [jobId, setActiveJobById])
```

**Flow:**
1. User clicks job in sidebar → URL becomes `/callum-dashboard?job=abc123`
2. Dashboard reads `job` query param
3. Calls `setActiveJobById('abc123')`
4. BaseContext queries database for job `abc123`
5. Extracts `base_id` from job record
6. All components re-render with new `base_id`
7. All queries now filter by new `base_id`

## Tenant Isolation Guarantees

### Level 1: Application Layer

All queries explicitly filter by `base_id`:
```typescript
.eq('base_id', baseId)
```

### Level 2: Context Layer

Supabase client carries `base_id` in request context:
```typescript
const client = createSupabaseClientWithContext(baseId)
```

### Level 3: Database Layer (Optional)

Row Level Security (RLS) policies can enforce tenant isolation:
```sql
CREATE POLICY "Users can only see their own job's profiles"
ON source_profiles
FOR SELECT
USING (base_id = current_setting('app.base_id')::text);
```

<Note>
The application uses **Levels 1 & 2** (application + context filtering). Level 3 (RLS) can be added for additional security in production environments.
</Note>

## Multi-Job Scenarios

### Scenario 1: Running Parallel Campaigns

**Setup:**
- Job A: Nike Instagram (base_id: `appNIKE123456789`)
- Job B: Adidas TikTok (base_id: `appADID987654321`)

**Operations:**

| Action | Job A | Job B | Isolation |
|--------|-------|-------|------------|
| Add source profile `@nike` | ✅ Saved with `appNIKE...` | ❌ Not visible | ✅ |
| Scrape 10,000 followers | ✅ Tagged `appNIKE...` | ❌ Not counted | ✅ |
| Check available usernames | ✅ 10,000 available | ✅ 0 available | ✅ |
| Create campaign | ✅ Uses Nike profiles | ❌ Cannot use Nike profiles | ✅ |
| Distribute to VAs | ✅ Nike's Airtable base | ✅ Adidas's Airtable base | ✅ |

### Scenario 2: Switching Between Jobs

**User Journey:**

<Steps>
  <Step title="User Viewing Job A">
    - URL: `/callum-dashboard?job=job-a-id`
    - BaseContext: `baseId = 'appNIKE...'`
    - Sees: Nike source profiles, Nike scraped data
  </Step>

  <Step title="User Clicks Job B in Sidebar">
    - URL changes to: `/callum-dashboard?job=job-b-id`
    - Dashboard detects URL change
    - Calls `setActiveJobById('job-b-id')`
  </Step>

  <Step title="BaseContext Updates">
    From `app/callum-dashboard/page.tsx:55-63`:
    ```typescript
    useResetOnChange(jobId, () => {
      console.log('Resetting state due to job ID change')
      setScrapedAccounts([])  // Clear old data
      setTotalFiltered(0)
      setRefreshKey(prev => prev + 1)  // Trigger re-fetch
    }, { skipInitial: true })
    ```
  </Step>

  <Step title="New Context Active">
    - BaseContext: `baseId = 'appADID...'`
    - All components re-query with new `base_id`
    - Sees: Adidas source profiles, Adidas scraped data
  </Step>
</Steps>

<Warning>
When switching jobs, **all local state is reset** to prevent showing stale data from the previous job. This includes scraped accounts, campaigns, and source profiles.
</Warning>

## API Integration with base_id

### Custom API Helper

From `lib/api.ts`:
```typescript
export async function apiPost(
  endpoint: string,
  baseId: string,  // Required tenant ID
  body: any
): Promise<any> {
  const response = await fetch(`${API_URL}${endpoint}`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Base-Id': baseId  // Pass in header for backend filtering
    },
    body: JSON.stringify({
      ...body,
      base_id: baseId  // Also include in body for redundancy
    })
  })
  
  return response.json()
}
```

### Backend Processing

Backend APIs receive `base_id` and use it to:
1. Filter database queries
2. Tag new records
3. Validate tenant access
4. Route to correct Airtable base

**Example: Daily Selection API**
```python
@app.route('/api/daily-selection', methods=['POST'])
def daily_selection():
    base_id = request.json.get('base_id')  # Extract tenant ID
    
    # Query only unused usernames for this tenant
    usernames = supabase.table('global_usernames') \
        .select('*') \
        .eq('base_id', base_id) \
        .eq('used', False) \
        .limit(14400) \
        .execute()
    
    # Create campaign tagged with base_id
    campaign = supabase.table('campaigns') \
        .insert({
            'base_id': base_id,
            'total_assigned': len(usernames.data)
        }) \
        .execute()
```

## Best Practices

### For Developers

1. **Always require base_id**
   ```typescript
   // ✅ Good
   if (!baseId) {
     throw new Error('base_id required')
   }
   
   // ❌ Bad
   const baseId = baseId || 'default'  // Never use defaults!
   ```

2. **Use context-aware clients**
   ```typescript
   // ✅ Good
   const client = createSupabaseClientWithContext(baseId)
   
   // ❌ Bad
   import { supabase } from '@/lib/supabase'  // Missing context
   ```

3. **Belt-and-suspenders filtering**
   ```typescript
   // ✅ Good
   const client = createSupabaseClientWithContext(baseId)
   const { data } = await client
     .from('source_profiles')
     .eq('base_id', baseId)  // Explicit filter + context
   
   // ⚠️ Risky
   const { data } = await client
     .from('source_profiles')  // Context only, no explicit filter
   ```

4. **Log base_id in operations**
   ```typescript
   console.log('[Component] Operating with base_id:', baseId)
   ```

### For Users

1. **Always select a job first**
   - Check sidebar shows active job
   - Verify URL has `?job=` parameter
   - Dashboard shows job info banner

2. **Watch for context switches**
   - UI resets when switching jobs (expected)
   - Data from previous job disappears (expected)
   - Always refresh after job switch

3. **Verify Airtable base**
   - Each job links to specific Airtable base
   - Check base_id matches in both systems
   - Don't manually copy data between bases

## Troubleshooting Multi-Tenant Issues

### "No active job" errors

**Cause:** BaseContext doesn't have an active `base_id`.

**Debug:**
```typescript
const { baseId, activeJob, isLoading } = useBase()
console.log({ baseId, activeJob, isLoading })
```

**Solutions:**
1. Select a job from the sidebar
2. Navigate to a job-specific URL: `/callum-dashboard?job={job_id}`
3. Create a new job if none exist

### Data from wrong job appearing

**Cause:** Stale `base_id` or missing tenant filter.

**Debug:**
```typescript
// Check what base_id is being used
console.log('Current base_id:', baseId)

// Check database records
SELECT DISTINCT base_id FROM source_profiles;
```

**Solution:**
1. Verify query includes `.eq('base_id', baseId)`
2. Check BaseContext shows correct job
3. Clear browser cache and reload

### Profiles not loading

**Cause:** Query filtering by wrong `base_id`.

**Debug:**
```sql
-- Check if profiles exist for this base_id
SELECT COUNT(*) 
FROM source_profiles 
WHERE base_id = 'appYOURBASEID';
```

**Solution:**
1. Verify profiles were saved with correct `base_id`
2. Check `base_id` matches between job and profiles
3. Re-add profiles if necessary

## Security Considerations

### Row Level Security (RLS)

For production deployments, enable RLS on all tables:

```sql
-- Enable RLS
ALTER TABLE source_profiles ENABLE ROW LEVEL SECURITY;

-- Create policy
CREATE POLICY "Tenant isolation for source_profiles"
ON source_profiles
FOR ALL
USING (base_id = current_setting('app.base_id')::text);
```

### API Authentication

Always validate `base_id` ownership:

```python
def validate_base_access(user_id, base_id):
    job = supabase.table('scraping_jobs') \
        .select('*') \
        .eq('base_id', base_id) \
        .eq('owner_id', user_id) \
        .single() \
        .execute()
    
    if not job.data:
        raise Unauthorized('Access denied to this base_id')
```

### Environment Variables

Never hardcode `base_id` in environment variables:

```bash
# ❌ Bad - hardcoded tenant ID
DEFAULT_BASE_ID=app123456789

# ✅ Good - fetch from database
SUPABASE_URL=https://xxx.supabase.co
SUPABASE_KEY=xxx
```

## Migration Guide

### Adding base_id to Existing Tables

<Steps>
  <Step title="Add Column">
    ```sql
    ALTER TABLE your_table 
    ADD COLUMN base_id TEXT;
    ```
  </Step>

  <Step title="Backfill Data">
    ```sql
    -- If you have a default job
    UPDATE your_table 
    SET base_id = 'appDEFAULT123456' 
    WHERE base_id IS NULL;
    ```
  </Step>

  <Step title="Add NOT NULL Constraint">
    ```sql
    ALTER TABLE your_table 
    ALTER COLUMN base_id SET NOT NULL;
    ```
  </Step>

  <Step title="Add Index">
    ```sql
    CREATE INDEX idx_your_table_base_id 
    ON your_table(base_id);
    ```
  </Step>

  <Step title="Update Application Code">
    ```typescript
    // Add base_id to all queries
    const { data } = await supabase
      .from('your_table')
      .select('*')
      .eq('base_id', baseId)  // New filter
    ```
  </Step>
</Steps>
