## Description

Syncs distributed profiles to Airtable VA workspace, pushing 180 profiles to each of 80 VA tables. This is the final step in the campaign workflow.

This endpoint:
1. Retrieves campaign assignments from `daily_assignments` table
2. Groups profiles by VA table number
3. Pushes profiles to corresponding Airtable tables (Table 1 - Table 80)
4. Updates campaign status to `success` or `failed`
5. Returns sync statistics

<Info>
  Airtable API rate limits are handled automatically with built-in delays between batch uploads.
</Info>

## Airtable Integration

### Required Configuration

Set Airtable credentials in backend environment:

```bash .env
AIRTABLE_API_KEY=your-airtable-key
AIRTABLE_BASE_ID=your-base-id
```

### Table Structure

Airtable workspace must have 80 tables named:
- `Table 1`
- `Table 2`
- ...
- `Table 80`

Each table should have these fields:
- `Username` (Single line text)
- `Profile URL` (URL)
- `Assigned Date` (Date)
- `Status` (Single select: Pending, In Progress, Completed)

### Rate Limiting

Airtable API has the following rate limits:
- **5 requests per second** per base
- **100,000 requests per day** per workspace

The sync endpoint handles rate limiting with:
- Built-in delays (200ms between requests)
- Automatic retry on rate limit errors (429)
- Batch processing (10 records per request)

## Sync Logic

Profiles are synced using batch processing:

```typescript
// Pseudo-code for Airtable sync
const syncToAirtable = async (campaignId: string, baseId: string) => {
  // 1. Get assignments grouped by VA table
  const assignments = await getAssignmentsByVA(campaignId)

  // 2. Sync each VA table
  for (const [vaNumber, profiles] of assignments) {
    const tableName = `Table ${vaNumber}`
    
    // 3. Batch profiles in groups of 10
    const batches = chunkArray(profiles, 10)
    
    for (const batch of batches) {
      // 4. Create Airtable records
      await airtable.base(baseId)(tableName).create(
        batch.map(p => ({
          fields: {
            Username: p.username,
            'Profile URL': `https://instagram.com/${p.username}`,
            'Assigned Date': new Date().toISOString(),
            Status: 'Pending'
          }
        }))
      )
      
      // 5. Delay to respect rate limits
      await delay(200)
    }
  }
}
```

## Campaign Workflow

This endpoint is the final step in the campaign workflow:

```typescript
const runCampaignWorkflow = async () => {
  // Step 1: Create campaign and select profiles
  const selectionResponse = await apiPost('/api/daily-selection', baseId)
  const { campaign_id } = selectionResponse

  // Step 2: Distribute to VA tables
  await apiPost(`/api/distribute/${campaign_id}`, baseId)

  // Step 3: Sync to Airtable (final step)
  const syncResponse = await apiPost(
    `/api/airtable-sync/${campaign_id}`,
    baseId
  )

  if (syncResponse.success) {
    console.log('Campaign completed successfully!')
    console.log(`Synced ${syncResponse.total_records} profiles to Airtable`)
  }
}
```

## Performance Considerations

<Warning>
  Airtable sync is the slowest operation in the workflow:
  - **80 tables** × **18 batches** = 1,440 API requests
  - With 200ms delays: ~5 minutes total
  
  Display progress updates to users during sync.
</Warning>

**Optimization Tips:**
- Use batch sizes of 10 records (Airtable maximum)
- Implement exponential backoff on rate limit errors
- Parallelize syncs across multiple tables (with caution)
- Monitor Airtable API quota usage

## Error Handling

Common error scenarios:

### Invalid Airtable Credentials

**Cause:** Invalid API key or base ID

**Solution:** Verify `AIRTABLE_API_KEY` and base ID in environment variables

### Table Not Found

**Cause:** Airtable base missing required tables (Table 1 - Table 80)

**Solution:** Create all 80 tables in Airtable base

### Rate Limit Exceeded

**Cause:** Too many requests in short period

**Solution:** Endpoint automatically retries with exponential backoff

### Field Mismatch

**Cause:** Airtable table schema doesn't match expected fields

**Solution:** Ensure tables have `Username`, `Profile URL`, `Assigned Date`, `Status` fields

## Retry Logic

The endpoint implements automatic retry on transient failures:

```typescript
// Pseudo-code for retry logic
const syncTableWithRetry = async (
  tableName: string,
  profiles: Profile[],
  maxRetries = 3
) => {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      await syncTable(tableName, profiles)
      return { success: true }
    } catch (error) {
      if (error.statusCode === 429) {
        // Rate limit - exponential backoff
        await delay(1000 * Math.pow(2, attempt))
        continue
      }
      throw error
    }
  }
  throw new Error(`Failed after ${maxRetries} retries`)
}
```

## Cleanup After Sync

After successful sync, the campaign enters a 7-day lifecycle:

1. VAs access profiles in Airtable
2. VAs mark profiles as "In Progress" or "Completed"
3. After 7 days, `/api/cleanup` endpoint removes old campaigns
4. Profiles are marked as unused and returned to the pool
