## Overview

The `scraping_jobs` table is the **core table** for multi-tenant job management. Each job represents an influencer's scraping campaign on a specific platform (Instagram, Threads, TikTok, or X).

<Info>
The `base_id` column in this table serves as the **source of truth** for multi-tenant isolation across all related tables.
</Info>

## Schema Definition

```sql
CREATE TABLE scraping_jobs (
  job_id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  influencer_name TEXT NOT NULL,
  platform TEXT NOT NULL,
  airtable_base_id TEXT,
  base_id TEXT NOT NULL,  -- Multi-tenant isolation key
  num_vas INTEGER,
  status TEXT DEFAULT 'active',
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

## Indexes and Constraints

```sql
-- Primary key index
CREATE INDEX idx_scraping_jobs_pkey ON scraping_jobs(job_id);

-- Platform filtering
CREATE INDEX idx_scraping_jobs_platform ON scraping_jobs(platform);

-- Status filtering
CREATE INDEX idx_scraping_jobs_status ON scraping_jobs(status);

-- Multi-tenant isolation
CREATE INDEX idx_scraping_jobs_base_id ON scraping_jobs(base_id);

-- Status constraint
ALTER TABLE scraping_jobs
  ADD CONSTRAINT scraping_jobs_status_check
  CHECK (status IN ('active', 'paused', 'archived'));

-- Platform constraint
ALTER TABLE scraping_jobs
  ADD CONSTRAINT scraping_jobs_platform_check
  CHECK (platform IN ('instagram', 'threads', 'tiktok', 'x'));
```

## Relationships

### One-to-Many: Job → Usernames

```typescript
// One job has many usernames
const usernames = await supabase
  .from('global_usernames')
  .select('*')
  .eq('job_id', jobId)
```

### One-to-Many: Job → Assignments

```typescript
// One job has many assignments
const assignments = await supabase
  .from('daily_assignments')
  .select('*')
  .eq('job_id', jobId)
```

### One-to-Many: Job → Campaigns

```typescript
// One job can have multiple campaigns
const campaigns = await supabase
  .from('campaigns')
  .select('*')
  .eq('base_id', job.base_id)
```

## Common Queries

### Get Active Job

```typescript
import { getActiveJob } from '@/lib/scraping-jobs'

const activeJob = await getActiveJob()
if (activeJob) {
  console.log('Active job:', activeJob.influencer_name)
  console.log('Base ID:', activeJob.base_id)
}
```

### Get Jobs by Platform

```typescript
import { getJobsByPlatform } from '@/lib/scraping-jobs'

const instagramJobs = await getJobsByPlatform('instagram')
const activeInstagramJobs = await getJobsByPlatform('instagram', true)
```

### Create New Job


## Multi-Tenant Workflow

### 1. Job Creation

```typescript
// User creates a new job via the Configure page
const job = await createScrapingJob({
  influencer_name: 'Taylor Swift',
  platform: 'instagram',
  airtable_base_id: 'appXYZ123',
  num_vas: 80
})

// base_id is automatically set to airtable_base_id
console.log(job.base_id) // "appXYZ123"
```

### 2. Context Initialization

```typescript
import { createSupabaseClientWithContext } from '@/lib/supabase'

// Create a context-aware client for this job
const client = createSupabaseClientWithContext(job.base_id)

// All queries use this client for proper isolation
const { data } = await client
  .from('global_usernames')
  .select('*')
  // RLS automatically filters by base_id from headers
```

### 3. Data Isolation

All related operations automatically filter by `base_id`:

```typescript
// Adding usernames to job
await addUsernamesToJob(jobId, baseId, usernames)

// Creating assignments
await createJobAssignments(jobId, campaignId, baseId, assignments)

// Fetching statistics
await getJobStatisticsByBaseId(baseId)
```

## Best Practices

<Check>**Always use base_id for multi-tenant operations** - Pass base_id explicitly to all functions that modify data</Check>

<Check>**Fetch active job on app load** - Use `getActiveJob()` to determine which base_id to use</Check>

<Check>**Validate base_id before operations** - Ensure base_id exists before creating assignments or usernames</Check>

<Warning>**Never hardcode base_id values** - Always fetch from the active job or user context</Warning>

## Legacy Placeholder Job

For backward compatibility with data created before multi-tenancy:

```typescript
export const LEGACY_PLACEHOLDER_JOB_ID = '00000000-0000-0000-0000-000000000001'

export function isLegacyJob(jobId: string | null): boolean {
  return jobId === LEGACY_PLACEHOLDER_JOB_ID
}
```

<Note>
The legacy placeholder job contains all data created before the multi-tenant migration. New jobs should never use this ID.
</Note>
